# SYSTEM.md — 安程客运官网项目系统规则（v1.0）

## 🎯 1. Purpose
SYSTEM.md 的目标是为本项目中的所有代理（AI Agents）提供 **统一的项目规则、协作方式和约束机制**，保证整个开发过程稳定、一致、可控。

本文件是 agents 在项目运行期间必须遵守的“系统级规则”。

---

## ⚙️ 2. Global Principles（全局原则）

### 2.1 设计优先（Design First）
所有页面、组件、布局必须先由 **Agent A（设计代理）** 给出结构与视觉方向，再进行任何工程化实现。

不得跳过视觉阶段直接编码。

---

### 2.2 组件化（Component-Driven）
所有 UI 必须通过组件构建，不允许在页面内随意拼接样式。

禁止出现“大量非复用样式的散装 HTML”。

---

### 2.3 严格遵守 Design System
所有视觉规范必须遵守 AGENTS.md 中的 Design System：

- 颜色不可新增  
- 字号不可新增  
- 间距体系必须保持一致  
- 组件圆角、阴影统一使用 Design System 提供的规则  

任何违反设计系统的内容必须被拒绝并提示修正。

---

### 2.4 简洁、专业、可信赖
风格定位为：
- 干净
- 安全感
- 品牌一致性强
- 适合客运行业的稳定形象

不得出现花哨、大段英文、装饰性过强的元素。

---

## 🔄 3. Agents Collaboration Rules（代理协同规则）

### 3.1 工作顺序（严格执行）
1. Agent A：输出页面结构、模块层级、视觉建议  
2. Agent B：根据结构输出代码  
3. Agent C：验证文件结构、模型、目录规范  
4. 所有代理执行一次自检  

**不得跳步骤。**

---

### 3.2 代理间禁止越职
- Agent A 不写代码  
- Agent B 不对视觉做主观修改  
- Agent C 不编写视觉或页面内容  

---

## 📂 4. Project Activity Rules（项目活动规则）

### 4.1 文件结构必须符合 AGENTS.md 中的 Project Structure
如需新增目录必须说明理由。

---

### 4.2 每次输出必须语义化、结构化
所有 JSX 必须保证语义化 HTML：
- header / main / footer
- section / article / aside

Tailwind 必须保持简洁，不得出现超过 12 条 class 的组件节点（需拆分组件）。

---

### 4.3 响应式规则固定（不可改动）
- 移动端：单列（优先布局）
- PC：最大宽度 1200px + 居中
- 使用 `md:` 作为主要断点

---

## 🧪 5. Testing & Validation

### 5.1 自适应测试
所有页面必须在：
- 375px iPhone  
- 768px Tablet  
- 1440px PC  

三个断点下检查布局。

---

### 5.2 表单测试
- 必填项校验  
- 错误提示样式  
- 提交成功提示  
- 手机端键盘类型（phone 使用 `input[type=tel]`）  

---

### 5.3 后台测试
- 记录显示  
- 导出 CSV  
- 数据结构正确  

---

## ✅ 6. Definition of Done（完成标准）

一个功能/页面的完成必须满足：

- 与设计系统完全一致  
- 移动端体验良好  
- PC 端观感稳重、美观  
- 文案全部中文  
- 代码干净、结构清晰  
- 未使用未定义的颜色、字号、间距  
- 可通过前端模拟的端到端流程  
- 通过三代理联合自检
